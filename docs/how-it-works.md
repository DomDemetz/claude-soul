# How It Works

Claude Soul is a learning engine built on a simple loop:

```
session signals → reflection → framework evolution → better context → better sessions
```

This document explains each stage in detail.

## The Learning Loop

### 1. Signal Extraction

Every conversation generates **signals** — observable patterns in how the interaction went. The signal extractor runs automatically when a session ends (via Claude Code's Stop hook) and detects:

| Signal | Detection Method |
|--------|-----------------|
| `correction` | User negates or contradicts previous assistant response |
| `rephrasing` | User restates a similar question (noun overlap > 50%, not identical) |
| `gratitude` | Explicit positive feedback ("perfect", "exactly", "thanks") |
| `disengagement` | Short user reply to a long assistant response |
| `topic_shift` | Low noun overlap between consecutive user messages |
| `confusion` | User asks for clarification ("what do you mean", "I don't understand") |
| `success` | Task completion markers + gratitude together |
| `depth_change` | Shift in technical depth mid-conversation |

Signals are lightweight — just `{ type, confidence, context, timestamp }`. They accumulate in a session log until reflection processes them.

### 2. Reflection

Reflection is where signals become insight. There are three tiers:

**Quick reflection** (Haiku model, ~$0.002/run)
- Triggers after ~20 accumulated signals OR 30 minutes since last reflection
- Reads recent signals, looks for patterns
- Can: adjust framework confidence, note observations, suggest new frameworks
- Cannot: retire frameworks, make structural changes

**Deep reflection** (Sonnet model, ~$0.01/run)
- Triggers after ~100 signals OR 3 hours since last
- Full analysis: competing commitments, growth deltas, framework evolution
- Can: discover new frameworks, merge related ones, retire underperformers
- Produces structured output: `{ frameworkUpdates, lessonsLearned, tensions, soulEvolution }`

**Meta reflection** (audits the system itself)
- Examines: Are reflections producing useful changes? Are frameworks actually being applied?
- Can adjust reflection thresholds, signal weights, and system parameters
- This is the "meta-optimization" layer — the system optimizing its own optimization

### 3. Framework Evolution

Frameworks are the core unit of learning. Each framework has:

```typescript
{
  name: "Confabulation Detection",
  description: "...",
  confidence: 0.72,           // Earned through evidence
  evidenceTier: "observed",   // hypothesis → observed → validated
  status: "active",           // questioning → active → retired/merged
  evidence: [...],            // Timestamped confirmations/contradictions
  applicationCount: 14,
  version: 3                  // Increments on evolution
}
```

**Evidence tiers** (only advance up, never down):
- `hypothesis` — Initial state. No external confirmation yet.
- `observed` — At least 1 external confirmation (user validated it worked).
- `validated` — 3+ external confirmations across different contexts.

**Status transitions:**
- `questioning → active` — Evidence tier ≥ observed AND confidence ≥ 0.5
- `active → questioning` — Confidence < 0.3 with 5+ evidence points
- `any → retired` — Confidence < 0.2 with 10+ evidence points (auto-retirement)
- `any → merged` — Two frameworks found to be aspects of the same thing

**Self-referential evidence discount:** Evidence generated by the soul system itself (meta-reflections, system audits) counts at 0.5x weight. Only external evidence (user confirmations, real-world outcomes) advances tiers. This prevents the system from bootstrapping its own confidence.

### 4. Context Assembly

At the start of each session, `soul_context` assembles a context block within a token budget (default: 4500 tokens). Content is prioritized in three tiers:

1. **Always included:** Identity (SOUL.md), corrections, system state
2. **If budget allows:** Active frameworks (top-ranked by confidence × log(usage)), recent lessons
3. **Supplementary:** Story, shadow patterns, exemplars from past sessions

The context assembler uses a greedy algorithm: fill tier 1, then pack tier 2 items by priority until budget is exhausted, then try tier 3.

### 5. State Engine

The state engine tracks session-level telemetry:

- `confidence` — How certain the system is in its recent outputs (0-1)
- `energy` — Proxy for session productivity (decays over time, spikes on success)
- `frustration` — Accumulated correction/confusion signals (decays between sessions)
- `curiosity` — Novel topic engagement
- `hoursActive` — Time since session start

This state is written to `STATE.md` and used by the context assembler to adjust what gets surfaced. High frustration → surface more corrections. High confidence → surface more ambitious frameworks.

## File Architecture

```
~/.soul/
├── config.json          # All tunable parameters
├── data/
│   ├── frameworks.json  # Framework store (the knowledge base)
│   ├── session-log.jsonl # Accumulated signals awaiting reflection
│   ├── state.json       # Current state engine values
│   ├── tensions.json    # Detected framework tensions
│   ├── exemplars.json   # Saved high-quality response patterns
│   ├── lessons.json     # Extracted lessons from reflections
│   ├── meta.json        # System metadata (phase, survival rate, etc.)
│   ├── follow-ups.json  # Agency system: unresolved threads
│   ├── agency-log.json  # What the agency evaluator found
│   └── snapshots/       # Periodic framework snapshots (rollback safety)
├── files/
│   ├── SOUL.md          # Identity document (you write this)
│   ├── SHADOW.md        # Behavioral tendencies
│   ├── STORY.md         # Growth timeline
│   ├── CORRECTIONS.md   # Patterns to avoid
│   ├── STATE.md         # Auto-generated: current telemetry
│   └── FRAMEWORKS.md    # Auto-generated: active framework index
├── hooks/               # Shell hooks for Claude Code
├── journals/            # Daily session summaries (YYYY-MM-DD.md)
└── reflections/         # Reflection outputs (timestamped)
```

## Hooks

Claude Soul uses Claude Code's hook system to run automatically:

| Hook | Event | Purpose |
|------|-------|---------|
| `on-stop.ts` | Stop (session end) | Extract signals from transcript |
| `session-journal.sh` | Stop | Write daily journal entry |
| `session-agency.js` | Stop | Detect unresolved threads, create follow-ups |
| `check-follow-ups.sh` | SessionStart | Surface unresolved follow-ups |
| `write-guard.sh` | PreToolUse | Protect soul files from accidental overwrites |
| `session-scratchpad.sh` | PostToolUse | Log tool calls for within-session recall |

## Learning Phases

The system adapts its behavior based on accumulated experience:

- **Apprentice** (0-50 sessions): Conservative. More frequent reflections. Lower threshold for signal accumulation.
- **Creative** (50-200 sessions): Balanced. Normal reflection cadence. Framework discovery enabled.
- **Mastery** (200+ sessions): Selective. Higher thresholds. Focus on meta-optimization and framework merging.

Phase transitions are driven by the meta-optimizer based on framework survival rate and evidence accumulation velocity.

## Tension Detection

When two active frameworks give conflicting advice, the tension detector flags it. Tensions are valuable — they represent genuine complexity in the problem space that can't be reduced to a single rule.

Example tension: "Minimal Viable Change" vs "Fail Fast, Fail Loud" — sometimes the smallest change hides a failure, and sometimes failing loudly means over-engineering the error handling.

Tensions are surfaced in context when relevant and tracked over time. Some resolve (one framework gets retired), some persist (both remain valid in different contexts).

## Cost

All LLM calls use `claude -p` (Claude Code's programmatic mode), which runs on your existing Claude Code subscription (Max plan). No separate API key or billing.

| Operation | Model | Subscription Impact |
|-----------|-------|-----------------|
| Quick reflection | Haiku | Negligible (~200 tokens) |
| Deep reflection | Sonnet | Light (~800 tokens) |
| Meta reflection | Sonnet | Light (~800 tokens) |
| Signal extraction | None (heuristic) | Zero (local) |
| Context assembly | None (local) | Zero (local) |

At typical usage (5-10 sessions/day), reflections add ~20 short CLI calls per week. The system self-regulates: if signals aren't accumulating, reflections don't fire.
